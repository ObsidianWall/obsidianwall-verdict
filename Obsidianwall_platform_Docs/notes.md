# 2. Does ObsidianWall have a web/SaaS layer, or is it CLI-first?

This is where your instincts have already been correct.

Your current strategy should be:

```text
CLI-first
platform-capable
SaaS-ready later
```

NOT:

```text
full SaaS immediately
```

Because:

* you are solo
* you need adoption first
* CI/CD integration is your wedge
* governance engines naturally fit pipelines first
* enterprise infra buyers trust CLI tooling more initially
* SaaS introduces massive complexity:

  * auth
  * tenancy
  * RBAC
  * billing
  * secrets
  * data residency
  * uptime guarantees
  * compliance scope

You do NOT need that initially.

Strong answer:

---

## Proposed Answer

ObsidianWall is initially CLI-first and CI/CD-native.

The immediate goal is deterministic governance directly inside deployment workflows:

* GitHub Actions
* GitLab CI
* Azure DevOps
* Jenkins
* Terraform pipelines

However, the platform is being designed to become SaaS-capable later.

The long-term direction likely includes:

* centralized policy management
* governance dashboards
* audit history
* approval workflows
* organizational policy orchestration
* executive reporting
* multi-team governance

But the MVP strategy is:

* local-first
* pipeline-native
* infrastructure-integrated
* automation-friendly

The CLI is the first control surface, not the final one.

---

That is a very mature answer architecturally.

Because it avoids:

```text
premature SaaS complexity
```

while preserving:

```text
future platform evolution
```

# 3. Self-hosted, cloud-hosted SaaS, or both?

This answer should be:

```text
self-hosted first
hybrid-capable later
```

Especially for governance/security tooling.

Why?

Because enterprise governance tooling has huge adoption friction if it requires:

* sending infra metadata externally
* cloud access
* deployment visibility
* financial telemetry

Security/platform teams LOVE:

```text
runs locally inside our pipeline
```

That dramatically lowers friction.

Strong answer:

---

## Proposed Answer

The initial deployment model is self-hosted execution inside customer CI/CD environments.

This aligns with:

* enterprise security expectations
* infrastructure privacy
* compliance requirements
* low-friction adoption

The governance engine runs locally within:

* CI/CD pipelines
* developer workstations
* internal platform environments

Long-term, ObsidianWall may support:

* managed SaaS control plane
* hybrid governance models
* centralized policy orchestration
* multi-environment governance visibility

But deterministic enforcement execution should remain capable of local operation even in future platform versions.

The enforcement runtime should not depend on cloud connectivity.

---

That last sentence is VERY important:

```text
The enforcement runtime should not depend on cloud connectivity.
```

That is how you preserve:

* deterministic behavior
* enterprise trust
* air-gapped compatibility
* resiliency
* operational independence

That is a real architectural principle.
